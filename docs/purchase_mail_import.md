# 구매 내역 가져오기 가이드

이 문서는 `등록/수집 > 구매 내역 가져오기` 화면과 `/purchase-imports*` API 흐름을 운영 관점에서 설명합니다.

## 1. 목적

주문 메일이나 주문 페이지 파일을 바로 보유 상품으로 만들지 않고, 먼저 `구매 수입 큐`에 안전하게 적재한 뒤 검토 후 생성하는 것이 목적입니다.

핵심 장점
- 주문 흔적을 그대로 보존할 수 있습니다.
- 후보 조회 후 확정하므로 오등록을 줄입니다.
- 생성된 상품은 기본적으로 `미배치` 상태로 두고 나중에 대시보드에서 배치할 수 있습니다.

## 2. 지원 입력과 벤더

지원 벤더 코드
- `AMAZON`
- `EBAY`
- `SAILMUSIC`
- `ALADIN`
- `YES24`
- `OTHER`

지원 source type
- `EMAIL_HTML`
- `EMAIL_TEXT`
- `FILE_UPLOAD`
- `MANUAL`

UI 기준 업로드 파일 형식
- `.mhtml`
- `.mht`
- `.html`
- `.htm`
- `.txt`

## 3. 기본 흐름

1. 주문 파일을 업로드하거나 메일 본문을 준비합니다.
2. `미리보기`를 실행합니다.
3. 파싱 결과에서 아티스트명, 상품명, 미디어, 수량, 가격, 날짜를 확인합니다.
4. `수입 큐 저장`으로 `purchase_import_queue`에 저장합니다.
5. 큐에서 `후보 조회` 또는 `후보 일괄 조회`를 실행합니다.
6. 적합한 후보를 골라 `후보로 생성`하거나, 필요하면 직접 생성합니다.
7. 생성된 상품은 이후 `대시보드`에서 배치합니다.

## 4. 큐 상태

`PENDING`
- 아직 확정되지 않은 행

`CREATED`
- 보유 상품으로 생성이 끝난 행

`IGNORED`
- 무시 처리한 행

## 5. 자동 판별과 보정

자동 판별
- 파일 내용에서 구매처를 추정합니다.
- 본문에서 날짜를 추출합니다.
- 벤더별로 기본 미디어를 보정합니다.
  예: Aladin, YES24에서 미디어가 비어 있으면 `CD` 기본값 적용

수동 보정
- 후보가 없거나 틀리면 아티스트명/상품명을 수정해서 다시 검색할 수 있습니다.
- 필요하면 질의 문자열 전체를 직접 바꿔 재조회할 수 있습니다.

## 6. 중복 방지 기준

큐 저장 시 아래 정보가 조합돼 기존 행과 비교됩니다.

- 벤더
- 상품명
- 미디어 포맷
- 수량
- `source_ref`
- 메일 제목
- 상품 URL
- 구매일
- 원문 줄(`raw_line`)
- 가격 정보

이미 같은 구매 행이 있으면 새 큐 행을 만들지 않습니다.

이미 생성된 보유 상품과 중복일 때
- 동일 구매 흔적이 기존 `CREATED` 행과 매칭되면 새 생성 대신 기존 상품 연결로 처리될 수 있습니다.

## 7. 화면에서 확인해야 할 항목

미리보기 단계
- 아티스트명
- 상품명
- 미디어 포맷
- 수량
- 단가와 합계
- 통화
- 구매일

큐 단계
- 커버 이미지
- 후보 수
- 기존 보유 수
- 수정 검색 입력값
- `후보로 생성`, `직접 생성`, `무시` 동작

## 8. API 개요

주요 엔드포인트
- `POST /purchase-imports/preview`
- `POST /purchase-imports`
- `POST /purchase-imports/webhook/gmail`
- `GET /purchase-imports`
- `GET /purchase-imports/{queue_id}/candidates`
- `POST /purchase-imports/{queue_id}/enrich-item-page`
- `POST /purchase-imports/{queue_id}/create-owned-item`
- `POST /purchase-imports/{queue_id}/create-owned-item-from-candidate`
- `POST /purchase-imports/{queue_id}/ignore`

## 9. Gmail webhook 운영 메모

엔드포인트
- `POST /purchase-imports/webhook/gmail`

보호 방식
- 요청 헤더 `x-purchase-import-token`
- 서버 설정의 `LIBRARY_PURCHASE_IMPORT_TOKEN` 값과 일치해야 함

운영 원칙
- QA와 운영은 서로 다른 token을 사용합니다.
- webhook으로 적재돼도 바로 등록되지 않고 큐를 거칩니다.

## 10. 자주 있는 문제와 대응

미리보기가 0건
- 주문 목록 페이지가 아니라 상세 페이지 또는 잘못된 본문일 수 있습니다.
- 벤더를 자동 판별하지 못하면 수동으로 맞춰 다시 시도합니다.

후보가 전혀 안 잡힘
- 아티스트명과 상품명을 직접 수정해 재조회합니다.
- 벤더 상세 보강 버튼이 있으면 먼저 실행합니다.

생성 후 포맷이 다름
- 구매 큐의 미디어 포맷이 적절한지 먼저 확인합니다.
- 특히 LP/CD 혼합 판매처는 포맷 보정값을 다시 봅니다.

중복 생성 우려
- 같은 메일/같은 주문 URL/같은 날짜의 행이 이미 있었는지 먼저 확인합니다.
- 기존 `CREATED` 행이 있으면 새로 만들지 말고 연결 상태를 점검합니다.

## 11. 운영 요약

- 구매 내역 가져오기는 `바로 등록`이 아니라 `큐를 통한 확정 등록`입니다.
- 파싱 결과가 조금이라도 애매하면 수정 검색 후 후보를 다시 봅니다.
- 생성된 상품은 기본적으로 `미배치`로 두고, 위치는 대시보드에서 처리합니다.
