## 1. 운영 원칙

핵심 원칙
- 위치 운영은 `대시보드`에서 합니다.
- 메타 수정은 `검색/관리`에서 합니다.
- 신규 상품 입력은 `등록/수집`에서 합니다.
- 구조 관리, 예외 처리, 백업은 `운영/연계`에서 합니다.

역할별 시작점
- 관리자
  - `대시보드`
  - `검색/관리`
  - `소스 보강`
  - `등록/수집`
  - `운영/연계`
- 현장 운영자
  - `운영 홈`

작업 순서 기준
1. 상품을 먼저 등록하거나 가져옵니다.
2. 필요한 경우 `소스 보강` 또는 `후보 조회`로 메타를 치환합니다.
3. 내부 마스터를 정리합니다.
4. 마지막에 `대시보드`에서 장식장/칸 배치를 처리합니다.

## 2. 대시보드

목적
- 장식장과 칸 기준으로 실제 위치를 운영하는 메인 화면

언제 쓰는가
- 장식장에 꽂힌 순서를 볼 때
- 미배치 상품을 칸으로 이동할 때
- 칸 안 순서를 정리할 때
- 최근 잘못 이동한 상품을 원위치로 되돌릴 때

사용 순서
1. 장식장을 클릭합니다.
2. 층과 칸을 선택합니다.
3. `선반 / 썸네일 / 리스트` 중 보기 모드를 고릅니다.
4. 상품을 클릭해 선택합니다.
5. `맨 앞`, `맨 뒤`, `직전 위치 복구`, 다른 칸 클릭 이동을 사용합니다.
6. 우측 `이동 작업대`에서 `미배치` 또는 `검색 결과`를 가져와 칸으로 이동합니다.

주의
- 위치 정리는 이 화면에서 끝내는 것이 좋습니다.
- 메타 수정까지 같이 하려고 하지 않는 것이 운영상 안전합니다.

## 3. 운영 홈

목적
- 현장 응대용 조회와 `운영 도우미` 요약 확인

언제 쓰는가
- 고객이 특정 곡이나 상품을 찾을 때
- 현재 위치와 직전 위치를 빨리 확인해야 할 때
- 요청곡 로그를 남길 때

사용 순서
1. 곡명, 아티스트명, 상품명, 바코드, 견출지 ID로 조회합니다.
2. 검색어는 공백을 정리한 정규화 값으로 다시 맞춰집니다.
3. 결과 목록 위 `운영 도우미`에서 정규화 검색어, 상단 후보, 현재 위치를 먼저 확인합니다.
4. 상단 후보는 기존 결과 안에서만 고릅니다. 우선순서는 `바코드 정확 일치 -> 견출지 ID 정확 일치 -> 제목/아티스트 정확 일치 -> 현재 배치된 행 -> 첫 번째 행`입니다.
5. 현재 위치가 있으면 `장식장 열기`로 바로 해당 장식장 화면을 엽니다.
6. 현재 위치가 없으면 `미배치`로 요약되고 `장식장 열기`는 나타나지 않습니다.
7. 아래 결과 목록에서 현재 위치와 직전 위치를 다시 확인합니다.
8. 필요한 경우 요청곡을 등록합니다.
9. 응대가 끝나면 요청 상태를 갱신합니다.

주의
- `운영 도우미`는 기존 조회 결과를 다시 요약해 보여주는 읽기 전용 카드입니다.
- 조회 결과 외에 별도 검색 엔드포인트를 호출하지 않고 `/operator/catalog-search` 한 번의 응답만 요약합니다.
- 요약 카드가 결과 순서를 바꾸거나 새 후보를 만들지는 않습니다.
- 미배치 상품은 `현재 위치`에 `미배치`로 표시되고 `장식장 열기`는 숨겨집니다.
- 현장 운영 화면은 조회와 요청 로그 중심입니다.
- 메타 수정은 관리자 화면에서 처리합니다.

## 4. 검색/관리

목적
- 보유 상품 검색, 메타 수정, 내부 마스터 확인

언제 쓰는가
- 특정 상품의 상세 메타를 고쳐야 할 때
- 같은 마스터의 보유 상품 구성을 확인할 때
- 연계 상품이나 연계 굿즈를 추가할 때

사용 순서
1. 아티스트명, 상품명, 곡명, 카탈로그, 바코드로 검색합니다.
2. 대상 상품을 엽니다.
3. 상품 정보, 곡 정보, 커버, 연계 정보를 확인합니다.
4. 필요하면 `정렬 아티스트명`, `정렬 정책`, `권장 보관 규격` 같은 운영 필드를 함께 확인합니다.
5. 저장합니다.

주의
- 위치 변경은 여기서 하지 말고 `대시보드`로 넘어갑니다.
- 같은 앨범의 여러 미디어가 있다면 먼저 마스터 묶음 상태를 확인합니다.

## 5. 소스 보강

목적
- 이미 등록된 상품을 Discogs / ManiaDB / Aladin 후보로 보강

언제 쓰는가
- 커버가 없을 때
- 곡 정보가 비었을 때
- 카탈로그, 레이블, 발매일 같은 메타가 부족할 때
- 수동 등록만 해 둔 상품을 외부 메타와 연결할 때

사용 순서
1. `대상`을 고릅니다.
   - `미연결만`
   - `전체`
   - `기연결만`
2. `대상 불러오기`
3. `전체 후보 조회`
4. 자동 선택된 후보를 확인하거나 직접 후보를 고릅니다.
5. `선택 후보 일괄 교체`

주의
- 이 화면은 기존 상품의 메타를 보강하는 곳입니다.
- 신규 상품 생성은 `등록/수집`에서 처리합니다.

## 6. 등록/수집 > 간편 등록

목적
- 최소 정보만 빠르게 입력해 보유 상품을 먼저 생성

언제 쓰는가
- 대량 등록 초기에 속도를 우선할 때
- 메타가 아직 부족하지만 일단 소장 정보를 만들어야 할 때

사용 순서
1. 포맷, 수량, 아티스트명, 상품명을 입력합니다.
2. 필요하면 발매일, 레이블, 커버 URL, 보관 슬롯을 함께 입력합니다.
3. 저장합니다.
4. 이후 `API 메타 조회`나 `소스 보강`으로 메타를 채웁니다.

주의
- 장식장 배치보다 등록 속도를 우선하는 화면입니다.
- 장식장을 지정하지 않으면 미배치로 두고 나중에 `대시보드`에서 이동하는 흐름이 좋습니다.

## 7. 등록/수집 > 구매 내역

목적
- 주문 페이지 파일을 읽어 구매 상품을 `구매 수입 큐`로 가져오기

지원 대상
- Amazon 주문 페이지 파일
- eBay 구매 페이지 파일
- Sailmusic 메일/주문 표 파일

기본 흐름
1. 주문 페이지 파일(`.mhtml`, `.mht`, `.html`, `.htm`, `.txt`)을 업로드합니다.
2. `미리보기`로 파싱 결과를 확인합니다.
3. `수입 큐 저장`으로 큐에 적재합니다.
4. 아래 `구매 수입 큐`에서 `후보 조회` 또는 `후보 일괄 조회`를 실행합니다.
5. 맞는 후보를 선택한 뒤 `후보로 생성`합니다.

큐에서 확인할 것
- 커버
- 아티스트명
- 상품명
- 미디어
- 수량
- 가격
- 후보 수

주의
- 이 화면은 구매 흔적을 라이브러리 상품으로 바꾸는 중간 단계입니다.
- 바로 장식장에 넣지 말고, 먼저 미배치로 생성한 뒤 대시보드에서 배치합니다.
- 후보가 애매하면 `아티스트명 수정 검색`, `상품명 수정 검색`으로 직접 검색어를 고칩니다.

## 8. 등록/수집 > 대량 등록

목적
- CSV 기반 일괄 입력

언제 쓰는가
- 다량의 기존 소장품을 한 번에 넣을 때
- 바코드, Discogs release id, 위치 정보가 미리 정리돼 있을 때

사용 순서
1. CSV 파일을 선택합니다.
2. 필요하면 기본 카테고리와 메모를 입력합니다.
3. 업로드합니다.
4. 자동 확정되지 않은 항목은 `검수 큐`에서 처리합니다.

주의
- 자동 등록이 아니라 `검수 큐` 중심 흐름입니다.
- 위치 컬럼과 release id가 잘 들어가 있을수록 수작업이 줄어듭니다.

## 9. 등록/수집 > 마스터 정리

목적
- 같은 앨범의 여러 보유 상품을 하나의 내부 마스터로 묶기

언제 쓰는가
- LP, CD, 재발매, 수입반이 따로 흩어져 있을 때
- 예외 큐의 `마스터 없음`을 정리할 때

사용 순서
1. 외부 마스터를 검색합니다.
2. 마스터를 선택합니다.
3. 필요한 경우 해당 마스터의 버전을 확인합니다.
4. 보유 상품을 선택합니다.
5. 마스터에 묶습니다.

주의
- 같은 앨범인데 서로 다른 내부 마스터로 나뉘지 않도록 먼저 이 화면에서 정리하는 것이 좋습니다.
- `정렬 아티스트명`은 궁극적으로 마스터 기준으로 관리합니다.

## 10. 운영/연계 > 시스템 상태

목적
- 앱 생존 여부, 외부 접속, QA 상태 확인

확인 순서
1. 앱 상태
2. 외부 로그인 주소
3. `/health`
4. 최근 launchd 로그
5. QA 현황

문제 발생 시
1. `/health`가 `200`인지 확인합니다.
2. launchd 로그 경로를 확인합니다.
3. `./scripts/check_library_status.sh --short`를 실행합니다.

## 11. 운영/연계 > 장식장 / 슬롯

목적
- 장식장 구조와 비정형 슬롯 관리

장식장 등록
- 이름, 규격, 층 수, 칸 수, 정렬 정책을 정합니다.
- 정렬 정책은 장식장 단위입니다.
  - `아티스트/발매/앨범`
  - `등록 코드 순`

슬롯 관리
- 비정형 구조나 예외 칸만 직접 수정합니다.
- 기본 구조는 장식장 등록에서 일괄 생성하는 쪽이 좋습니다.

운영 팁
- 앞쪽 전시 장식장은 `아티스트/발매/앨범`
- 뒤쪽 증설/보관 장식장은 `등록 코드 순`
로 나누면 재배치량이 줄어듭니다.

## 12. 운영/연계 > 예외 큐

목적
- 병목 상품만 모아 빠르게 처리

우선순위
1. 미배치
2. 소스 미연결
3. 마스터 없음
4. 커버 없음
5. 보관 규격 예외
6. 곡 정보 없음

사용 순서
1. 예외 종류를 선택합니다.
2. 표시 수를 정합니다.
3. 필요한 경우 저장된 필터를 적용합니다.
4. 다중 선택 후 `소스 보강`, `실물 기준 맞춤`, `마스터 정리`로 보냅니다.

주의
- 이 화면은 “정답을 찾는 곳”이 아니라 “병목을 줄이는 곳”입니다.
- 정상 상품보다 예외 상품을 먼저 정리할수록 전체 작업 속도가 빨라집니다.

## 13. 운영/연계 > 계정

목적
- 관리자 / 현장 운영자 계정 관리

원칙
- 기본 시스템 계정은 읽기 전용
- 새 계정은 여기서 생성
- 권한에 따라 노출 화면이 달라짐

운영 팁
- 현장 운영자는 조회/요청 중심으로 두고, 관리자만 메타 수정 권한을 쓰는 것이 안전합니다.

## 14. 운영/연계 > 백업 / 내보내기

목적
- 대량 작업 전 보호 장치와 운영용 데이터 추출

권장 절차
1. 대량 수정 전 `DB 백업 다운로드`
2. 운영 공유용으로 `보유 상품 CSV`, `마스터 CSV`
3. 필요하면 수입 큐, QA 시트와 함께 보관

## 15. 운영/연계 > 메타 동기화

목적
- 이미 연결된 외부 소스 상품의 누락 메타 보강

언제 쓰는가
- 커버, 바코드, 카탈로그, 트랙리스트가 비어 있을 때
- 소스 보강 이후 후속 정리를 돌릴 때

기본 원칙
- `누락 필드만 보강`을 기본으로 둡니다.
- 전체 동기화는 필요할 때만 실행합니다.

## 16. 운영 한 줄 정리

- 위치는 `대시보드`
- 메타 수정은 `검색/관리`
- 메타 보강은 `소스 보강`
- 신규 입력은 `등록/수집`
- 구조/예외/백업은 `운영/연계`
